The Problem

Medication errors are the most common cause of patient harm in clinical settings — a missed dose or wrong administration can trigger life-threatening complications in chronic patients.

The Solution

Without a real-time tracking system, there is no safety net between a human mistake and a patient crisis. MedAlert closes that gap — providing live oversight, instant alerts, and a full audit trail for every dose administered.

Dependability — The 5 Attributes
Safety
Freedom from harm
  • Dose rejected if > MaxDose
  • Double-confirmation for medication deletion
  • System halts dose logging if DB unreachable
Security
Protected from intrusion
  • Role-based access (Nurse / Doctor / Admin)
  • Firebase Auth + Admin 2FA (TOTP)
  • HTTPS encryption in transit
Reliability
Delivers service without failure
  • Automated daily DB backups
  • Input validation on all entries
  • All exceptions caught and logged
Availability
Ready when requested
  • High availability 7AM–11PM clinic hours
  • Alerts queued outside clinic hours
  • Graceful error pages — no silent crashes
Resilience
Maintains service under failure
  • PM2 auto-restart on server crash
  • Rate limiting against brute-force
  • Full audit trail for post-incident recovery
Critical Note: Attributes are interconnected — a Security breach destroys Availability. MedAlert is designed with this interdependence in mind.

06
Lessons Learned
Concept — Dependability
Safety and Security are not the same thing
Before this course I treated "safe" and "secure" as synonyms. Working on MedAlert made the distinction concrete: Safety means the system cannot harm users even when it fails (e.g., halting dose logging if the DB goes down rather than guessing), while Security means unauthorized parties cannot access or corrupt the system. A system can be safe but insecure, or secure but unsafe. Our MaxDose enforcement is a safety measure; our RBAC is a security measure — they solve different problems.
Concept — Fault Tree Analysis
AND gates make systems safer than OR gates
When I drew the fault tree, I noticed that branches with AND gates require multiple simultaneous failures to trigger — meaning any one mitigation breaks the chain. This showed me why defense-in-depth works: not because any single layer is perfect, but because attackers or faults must overcome all layers at once. For MedAlert's "Incorrect Medication Administered" branch, both a nurse data entry error AND a DB sync failure must occur together. Fixing either one prevents the top event.
Concept — Architecture
3-Tier separation forces better engineering decisions
Separating the client, application, and data tiers forced us to think about where business logic lives. Early in the project I considered putting the MaxDose validation in the frontend (React) for a faster user experience. The 3-tier separation principle made it clear this was wrong — the frontend can be bypassed by a direct API call. The safety rule must live in the application tier where it cannot be circumvented. This is a concrete example of why architecture decisions have security and safety consequences.
